PR #412 · Sandbox optimization · 三模式统一模型

从早期混合沙箱，到新的沙箱和审批模型

这页用图解释 OpenSquilla 早期混合式沙箱方案和现在新模型的区别。 目标是让第一次看这个 PR 的人也能快速明白：什么是沙箱，什么时候需要审批， 为什么 Full Host Access 不应该再审批，以及谁能在频道里开启宿主机权限。

3 种运行模式 Standard-Sandbox、Trusted-Sandbox、Full Host Access。
三模式规则 Full Host 就是不沙箱、不审批；Sandbox 才审批。
频道默认可信沙箱 普通用户不处理沙箱审批，管理员可切 Full Host Access。

先记住这张总图

早期模型的问题是几套开关互相缠绕；新模型先确定运行模式，再自然决定要不要审批。

三种模式分别干什么

小白理解方式：从左到右，信任程度越来越高，限制越来越少。

Standard-Sandbox

最保守。工具在受限制环境里跑，碰到越界访问时需要人决定。

  • 适合陌生任务
  • 更强调隔离
  • 可能看到更多审批

Trusted-Sandbox

仍然有沙箱，但默认更顺滑。频道任务默认走这个模式。

  • 适合常规可信任务
  • 减少无谓打断
  • 仍保留边界

Full Host Access

不进沙箱，不走沙箱审批。只有 owner 或配置好的频道管理员能开。

  • 适合完全信任环境
  • 性能和兼容性最好
  • 风险也最高

和早期混合模型一眼对比

早期模型最大的问题不是“没有安全”，而是沙箱、工具审批、权限模式和频道审批叠在一起，让用户不知道自己到底在批准什么。

审批按钮变成三种

对小白来说，审批不是在理解系统内部状态，而是在回答一个简单问题：这次要不要放行？

频道里为什么不弹审批

频道消息不是一个可靠的审批面板。用户可能收不到通知，也可能无法正确 resolve。现在频道默认走 Trusted-Sandbox，必要时由管理员切换。

频道管理员怎么配置

在配置中把平台 sender id 加到对应 channel source 下。例如：

[channel_admin_senders]
feishu = ["sender-id-1"]
slack = ["sender-id-2"]

技术组件各自的作用

不用记代码，只要知道每个部件负责哪件事。

典型流程

下面两条路径是最常见的：沙箱里请求扩权，或者管理员直接开 full。

1. 任务进入 系统知道来源是 WebUI、CLI 还是频道。
2. 判断身份 owner 或频道管理员可以选择 full。
3. 选择运行模式 standard、trusted 或 full。
4. 工具执行 沙箱模式可能审批，full 直接宿主机执行。

改动清单

这里的“之前”指更早期的混合模型，不是最后几次 cleanup commit。

领域 早期混合模型 现在三模式统一模型
运行模式 沙箱、权限和审批像多套开关，用户难以判断当前是哪条路线在生效。 统一成 Standard-Sandbox、Trusted-Sandbox、Full Host Access。
Full Host 容易被理解成“更高权限，但可能仍然要审批”。 owner/admin-only；无沙箱、无沙箱审批。
审批 工具审批、沙箱审批、intent cache 的边界不够直观。 沙箱审批就是三选一：一次、同类型、拒绝。
频道 频道沙箱审批可能通知不到或无法正确 resolve。 频道默认 Trusted-Sandbox，不做聊天内沙箱审批，管理员可切 full。
沙箱开关 “沙箱关闭”仍可能和旧审批门纠缠。 沙箱关闭就是 Full Host Access，不再走沙箱审批。
