Squilla Tree Router Driven LLM Ensemble Planner

本文只讨论 llm_ensemble.enabled=true 的路径。目标是把本地 Squilla tree router 输出的 c0..c3 / image_model 档位，转成确定性的 proposer / aggregator 执行计划。Planner 不做任务类型理解，不假设存在 rich LLM Router JSON。

1. 设计边界

输入只有 tree router 强度信号 可依赖 routed_tier、routed_model、routing_confidence、routing_source 和 routing_extra。
所有 tier 都生成 ensemble plan c0 是轻量 ensemble，c3 是重型 ensemble；不在本文讨论其他执行路径。
Proposer 选择必须可解释 第一个 proposer 锚定 routed_model，后续 proposer 来自 operator 配置和 tier ladder。
不做任务类型专家路由 tree router 没有 intent.type、target.family、response.shape，所以第一版不做 code/math/agentic tag 匹配。
模型事实复用现有配置 候选池来自 llm_ensemble.profiles[llm_ensemble.active_profile]、squilla_router.tiers 和 ModelCatalog。

2. Router Metadata Contract

当前 V4 tree router 的核心返回形态是 (tier, confidence, source, extra)。 apply_squilla_router 会把这些值写入 turn metadata。

字段 Planner 用法
routed_tier 决定 proposer 数量和 tier ladder 顺序。
routed_model 作为 proposer anchor，优先放在 proposer 1。
routing_confidence / margin 决定是否增加一个 proposer，增强分歧覆盖。
routing_source 记录 tree router 输出来源，进入 plan trace；不扩展成额外任务标签。
本文术语 代码字段 含义
Router Anchor metadata.routed_model tree router 当前轮选出的主模型，固定作为第一个 proposer 候选。
Configured Proposer Pool llm_ensemble.profiles[llm_ensemble.active_profile].proposers operator 配置的 ensemble 候选池；顺序代表产品偏好和多样性优先级。
Configured Aggregator llm_ensemble.profiles[llm_ensemble.active_profile].aggregator operator 配置的默认综合模型，专门负责读取 proposer 输出并合成最终答案。
Aggregator Plan ensemble_plan.aggregator planner 选出的 synthesis member 和原因；这是 aggregator_plan 的序列化位置。
Tier Ladder squilla_router.tiers router tier 绑定的模型表；当候选池不够时提供确定性的补齐顺序。
Model Feature Catalog 新增 planner catalog，初始可由 llm_ensemble.model_options、squilla_router.tiers 和静态表合并生成。 给每个模型补充 family、tier bucket、quality bucket、cost bucket、latency bucket、context window、vision support 等静态特征。

本表只定义 ensemble 打开后的规模。每一行都会得到 proposer fan-out 和 aggregator synthesis。

Router tier Policy Proposers Aggregator 用途
c0 tier_c0_light 2 configured aggregator 低成本双候选，保留 ensemble 行为但控制规模。
c1 tier_c1_light 2 configured aggregator 常规对话和普通 agent work 的默认融合。
c2 tier_c2_standard 3 configured aggregator 复杂分析、多步编码、较大上下文综合。
c3 tier_c3_heavy 4 strong aggregator 高难度规划、深度审查、复杂 debug、高风险综合。
image_model vision_ensemble 2 vision-capable candidates vision-capable aggregator 只使用能消费图像输入的成员，不混入纯文本模型。
动态增强：c1 或 c2 在低 margin、 confidence_gate_applied=true 或 anti_downgrade_applied=true 时， proposer 数量可加 1，但不超过配置候选池上限。

4. Proposer Optimization Planner

这一节把 proposer 选择定义成 slot-local greedy scoring： 给定 router anchor 和一个较大的候选池，先固定 slot1=anchor， 再按每个 slot 自己的 target 和权重逐槽打分，而不是按列表顺序拿前几个。 这保留 planner 的可解释性，但特征来源换成本地 Model Feature Catalog。

输入

  • routed_tier：例如 c2。
  • routed_model：例如 z-ai/glm-5.2。
  • policy 要求的 proposer 数量：例如 c2 = 3。
  • Configured Proposer Pool：llm_ensemble.profiles[llm_ensemble.active_profile].proposers。
  • Model Feature Catalog：每个模型的 family、tier bucket、质量、成本、延迟、窗口和模态特征。

输出

  • 一个有顺序的 proposer 列表。
  • 每个 proposer 都带 source/reason，能解释它为什么被选中。
  • 列表长度必须满足 policy 要求。
  • 默认鼓励互补模型；允许同一 (provider, model) 再次入选，但会承受重复惩罚。
  • 输出 rank_trace，记录每个 slot 的候选分数和淘汰原因。
1

候选扩展

候选来自 Configured Proposer Pool、router anchor、tier ladder 和允许的 model options。 每个候选先转成统一的 feature vector。

2

硬约束过滤

先过滤掉 credential 不可用、窗口不够、模态不匹配的候选；重复模型不硬删，交给 objective 惩罚。

3

逐槽打分

每填一个 slot，就读取该 slot 的 target 和权重，对所有可执行候选重新打分。 质量、互补性、角色匹配、成本和延迟一起进入当前 slot 的目标函数。

关键点：当前 tree router 没有任务标签、领域标签或 rich JSON。 复杂度来自 planner 自己维护的静态模型特征，而不是伪造“本轮任务标签”。
术语 / invariant 含义
slot_target 当前 slot 希望补上的角色，例如 adjacent_tier_check 或 orthogonal_family；它是打分输入，不是模型名。
score 只在当前 slot 和当前权重向量下有意义。一个 scalar score 不能反推出 slot，也不能跨 slot 直接比较。
active_profile 只表示 operator 配置的候选来源；它不是 planner 输出的最终选择结果。
role_match 候选和当前 slot_target 的匹配度，解决“slot2 到底想找什么”的问题。
diversity_gain 候选相对已选 proposers 的互补性，解决“会不会都选同家族/同 provider”的问题。
Feature 来源 用途
quality_prior 静态 catalog / operator 配置 候选的基础质量档位，由静态配置给出，也不随当前 turn 伪造任务标签。
router_affinity routed_tier + candidate tier prior 衡量候选和当前 router 强度是否匹配，低 confidence 时会放宽。
diversity_vector model family、provider、tier bucket、vendor tag 降低 ensemble 内部同质化，优先补互补视角。
repeat_penalty 当前已选 proposer 列表 同一模型再次入选会扣分；如果扣分后仍是最高 objective score，则允许保留。
cost_latency_prior 静态 catalog / operator 配置 控制低 tier 的成本，避免轻量请求被过度放大。
Tier policy 的作用：它不直接指定模型名，而是给 slot-local scorer 三个东西： proposer_count、每个 slot 的 slot_spec、以及 base weights。 每次填 slot 时，scorer 会先取该 slot 的 target 和 weight overrides，再对所有候选重新打分。 
objective(c | selected, ctx, slot) =
    slot_spec = policy.slot_specs[slot]
    weights = merge(policy.base_weights, slot_spec.weight_overrides)

    + weights.w_quality        * quality_prior(c)
    + weights.w_affinity       * router_affinity(c, ctx.routed_tier)
    + weights.w_diversity      * diversity_gain(c, selected)
    + weights.w_role           * role_match(c, slot_spec.target)
    - weights.w_repeat         * repeat_penalty(c, selected)
    - weights.w_cost           * normalized_cost(c)
    - weights.w_latency        * normalized_latency(c)
    - weights.w_hard_penalty   * hard_penalty(c, ctx)
注意：最终只有一个 scalar score，但这个 score 是“针对当前 slot”算出来的。 同一个候选在 slot2=adjacent_tier_check 和 slot3=orthogonal_family 下会重新计算，分数可能不同。
Policy 权重倾向 实际效果
tier_c0_light 提高 w_cost、w_latency，降低 w_quality。 轻量请求优先选便宜快的对照模型。
tier_c1_light 质量、成本、延迟比较均衡。 普通请求补一个稳定对照，不主动拉满强模型。
tier_c2_standard 提高 w_affinity、w_diversity、w_role。 先找相邻 tier checker，再找跨 family 视角。
tier_c3_heavy 提高 w_quality、w_role，降低成本惩罚影响。 高难度请求更愿意选强审查模型和跨家族独立推理模型。
Slot role hint：下面这张表不是最终模型名单，也不是硬规则。 它只是告诉 slot-local scorer：不同 tier 的 ensemble 最好覆盖哪些“视角”。 如果某个候选匹配这些视角，role_match 会给它加分；如果重复模型扣分后仍更合适，也可以被选中。
Policy Proposer slot target 意思
tier_c0_light slot1=anchor, slot2=low_cost_contrast 第二个模型优先找便宜、快、且不要和 anchor 太像的候选。
tier_c1_light slot1=anchor, slot2=balanced_contrast 第二个模型优先找质量/成本比较平衡、能交叉验证 anchor 的候选。
tier_c2_standard slot1=anchor, slot2=adjacent_tier_check, slot3=orthogonal_family 第二个模型偏相邻 tier 做稳态对照，第三个模型偏不同家族提供独立视角。
tier_c3_heavy slot1=anchor, slot2=strong_critic, slot3=orthogonal_family, slot4=fast_sanity_check 高难度请求优先补强审查、跨家族独立推理，以及一个低延迟 sanity check。 
def select_proposers(ctx, config, proposer_count):
    tier = normalize_tier(ctx.metadata["routed_tier"])
    policy = policy_for_tier(tier, ctx.metadata)
    anchor = candidate_from_routed_model(ctx.metadata["routed_model"])
    pool = expand_candidate_pool(config, tier)

    selected = [enrich(anchor, config.model_feature_catalog)]
    candidates = [
        enrich(candidate, config.model_feature_catalog)
        for candidate in pool
        if hard_constraints_ok(candidate, ctx)
    ]

    rank_trace = []
    while len(selected) < proposer_count:
        slot = len(selected) + 1
        slot_spec = policy.slot_specs[slot]
        weights = merge_weights(policy.base_weights, slot_spec.weight_overrides)
        scored = [
            score_candidate(candidate, selected, ctx, slot_spec, weights)
            for candidate in candidates
        ]
        scored.sort(key=lambda item: (-item.score, item.config_index, item.model))
        if not scored:
            raise EnsembleConfigError("no executable proposer candidates")

        chosen = scored[0]
        selected.append(chosen.candidate)
        rank_trace.append({
            "slot": slot,
            "slot_target": slot_spec.target,
            "chosen": chosen.candidate.model,
            "score": chosen.score,
            "weights": weights,
            "components": chosen.components,
            "top_rejected": scored[1:4],
        })
        candidates = mark_selected_for_repeat_penalty(candidates, chosen.candidate)

    return selected, rank_trace

5. Proposer Filters

第 5 节定义硬过滤条件。实现上先过滤，再交给第 4 节的 ranker 逐槽选择。 过滤器不做质量排序，只判断候选能不能真的执行；重复模型不是硬过滤条件。

一句话：过滤器只回答 yes/no：这个候选是否可用？可用就保留，不可用就记录原因并继续拿下一个候选。
检查项 保留条件 不满足时
重复模型 允许同一 (provider, model) 再次作为候选。 不移除；在 objective 中应用 repeat_penalty。
上下文窗口 模型窗口能容纳原始输入和预估输出。 移除该候选，记录 context_window_too_small。
视觉输入 当前 turn 有图像时，候选必须支持 vision。 移除不能消费图像的候选。
Provider credential provider 能解析 API key/base URL。 移除该候选，记录 credential_missing。
候选不足 过滤后仍达到 policy 要求的 proposer 数量。 继续从第 4 节的候选队列取下一个；允许重复候选带惩罚竞争。
过滤例子 结果
google/gemini-3-flash-preview 没有可用 credential 从列表移除，继续取 Configured Proposer Pool 里的下一个候选，例如 qwen/qwen3.7-plus。
可用互补模型不足 重复候选不会被直接拒绝；如果扣除 repeat_penalty 后仍最高，就继续入选。
重复不是失败分支：planner 默认追求互补，但不把互异性当成硬合同。 同一模型重复入选必须能在 objective 里打过其它候选，并在 rank_trace 里暴露扣分。

6. Aggregator Optimization Planner

Aggregator 不参与“提出候选答案”的多样性竞争。它是一个单独角色：读取所有 proposer 输出， 做冲突消解、证据合并和最终格式化。因此 aggregator 用另一套 objective：综合能力、上下文容量、 成本、延迟和当前 proposer set 的输出规模。

候选来源

  • Configured Aggregator：llm_ensemble.profiles[llm_ensemble.active_profile].aggregator。
  • Strong Tier Aggregator：squilla_router.tiers.c3.model，用于高强度 synthesis。
  • Vision Aggregator：当前 turn 有图像时，从 vision-capable 候选里选。
  • Catalog Candidates：Model Feature Catalog 中标记为 synthesis-capable 的可用模型。

条件变量

  • proposer_count 和预计 candidate output 总长度。
  • routed_tier、routing_confidence、margin。
  • 是否有图像输入，以及 proposer 之间的预计分歧程度。
  • aggregator 的 cost、latency 和 context window。
aggregator_score(a | ctx, proposers) =
    + 0.35 * synthesis_quality_prior(a)
    + 0.20 * context_fit(a, ctx, proposers)
    + 0.15 * policy_affinity(a, ctx.routed_tier)
    + 0.10 * disagreement_capacity(a, proposers)
    + 0.05 * configured_stability_bonus(a)
    - 0.10 * normalized_cost(a)
    - 0.05 * normalized_latency(a)
    - 0.20 * hard_penalty(a, ctx)
场景 权重变化 效果
c0/c1/c2 文本请求 提高 configured_stability_bonus 和 cost 权重。 偏向稳定、便宜、窗口足够的综合模型。
c3 文本请求 提高 synthesis_quality_prior 和 disagreement_capacity 权重。 更容易选 strong tier aggregator，避免弱综合模型压平候选差异。
图像请求 hard_penalty 直接淘汰非 vision-capable aggregator。 只在能消费图像上下文的候选中评分。
默认配置映射：当前内置配置里，Configured Aggregator 是 z-ai/glm-5.2； Configured Proposer Pool 是 deepseek/deepseek-v4-pro、 z-ai/glm-5.2、google/gemini-3-flash-preview、 qwen/qwen3.7-plus。这不是 planner 推理出来的，是 operator 配置给 planner 的候选池。 
def resolve_aggregator(ctx, config, proposers):
    tier = normalize_tier(ctx.metadata["routed_tier"])
    pool = config.llm_ensemble.profiles[config.llm_ensemble.active_profile]
    candidates = [
        candidate_from_ref(pool.aggregator, reason="configured_aggregator"),
        candidate_from_tier("c3", reason="strong_tier_aggregator"),
        *catalog_synthesis_candidates(config.model_feature_catalog),
    ]
    if ctx.has_image:
        candidates.append(candidate_from_tier("image_model", reason="vision_tier_aggregator"))

    candidates = collapse_identical_aggregator_candidates(candidates)
    candidates = [
        enrich(candidate, config.model_feature_catalog)
        for candidate in candidates
        if synthesis_constraints_ok(candidate, ctx, proposers)
    ]
    scored = [
        score_aggregator(candidate, ctx, proposers)
        for candidate in candidates
    ]
    scored.sort(key=lambda item: (-item.score, item.config_index, item.model))

    if not scored:
        raise EnsembleConfigError("no viable aggregator")
    return scored[0].candidate, scored[0].components

8. Runtime 接入

Planner 插在 router metadata 已经产生之后、provider wrapper 构造之前。它只生成执行计划； EnsembleProvider 继续负责并发调用、candidate trace、aggregator synthesis 和 usage 汇总。

apply_squilla_router写入 routed_tier / routed_model
resolve_ensemble_policytier -> proposer_count
enrich_candidatescatalog features
score_proposer_slotsslot-local greedy scoring
score_aggregator_plansynthesis objective
ctx.metadata写入 ensemble_plan
EnsembleProviderfan-out + synthesis
实现建议：先把 planner 写成纯函数，输入 config、metadata、 token estimate，输出 plan dict。provider builder 再把 plan 转成 EnsembleMemberConfig。

9. 测试矩阵

Case 断言
c0 生成 2 proposers；第一个是 routed_model。
c2 生成 3 proposers；anchor 后用 objective score 逐槽选择，并输出 rank_trace。
8 个候选池，anchor 已占 1 个 剩余 7 个不随机；slot 2/3 都写入 score components、chosen 和 top rejected。
候选池同质化 diversity_gain 拉开同 family 模型，除非质量或成本差距明显。
低 margin / 低 confidence 提高 diversity_gain 和 role_match 权重，增加分歧覆盖。
c3 生成 4 proposers；aggregator objective 更偏向 synthesis quality 和 disagreement capacity。
配置候选池包含 routed model anchor 先入选；候选池里的同模型项继续保留，但带 repeat_penalty 参与竞争。
上下文窗口不足 过滤该候选并在 filtered_candidates 记录原因。
vision input 所有 proposer 和 aggregator 都满足 vision capability。
候选不足 允许重复模型带惩罚入选；只有没有可执行候选时才返回 ensemble_config_error。

10. 待定项

c3 proposer 数量是 3 还是 4？ 当前草案使用 4，以匹配默认配置候选池的 4 proposer 能力。
Model Feature Catalog 第一版放在哪里？ 建议先放在 gateway config 的默认表里，再允许 Settings UI 覆盖 quality/cost/latency prior。
objective 权重是否固定？ 当前草案用 tier policy 提供默认权重；后续如需调参，也先通过配置显式修改。
repeat penalty 应该多大？ 当前草案使用 same model 0.15、same family 0.08；如果重复模型扣分后仍最高，可以继续入选。
配置候选池顺序是否就是产品偏好？ 当前草案认为是。Settings UI 后续可以把 proposer 顺序做成显式可排序。
c3 aggregator 是否强制使用 strong tier？ 当前草案优先使用 squilla_router.tiers.c3.model，不可用时再使用 Configured Aggregator。
