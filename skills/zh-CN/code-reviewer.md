# 代码审查技能

## 技能说明

系统化审查代码，从质量、安全、性能、可维护性等多个维度提供专业反馈。

## 使用场景

| 场景 | 说明 |
|------|------|
| PR 审查 | 检查 Pull Request 代码质量 |
| 代码优化 | 识别可改进的代码模式 |
| 安全审计 | 检查潜在安全漏洞 |
| 性能分析 | 识别性能瓶颈 |
| 最佳实践 | 确保符合编码规范 |

## 审查维度

### 1. 代码质量

- **命名规范**：变量/函数/类名是否清晰
- **代码结构**：是否遵循单一职责原则
- **重复代码**：是否存在可抽取的重复逻辑
- **注释质量**：必要处是否有清晰注释
- **复杂度**：圈复杂度是否过高

### 2. 安全性

- **注入漏洞**：SQL/命令/LDAP 注入风险
- **XSS**：跨站脚本攻击风险
- **认证授权**：权限检查是否完整
- **敏感数据**：密钥/密码是否硬编码
- **输入验证**：用户输入是否充分验证

### 3. 性能

- **算法复杂度**：时间/空间复杂度是否最优
- **数据库查询**：是否存在 N+1 查询
- **缓存策略**：是否合理使用缓存
- **资源管理**：连接/文件是否正确释放
- **并发问题**：是否存在竞态条件

### 4. 可维护性

- **模块化**：代码是否易于测试和修改
- **依赖管理**：依赖是否合理且最少
- **错误处理**：异常处理是否完整
- **日志记录**：关键路径是否有日志
- **版本兼容**：是否考虑向后兼容

## 工作流程

### Step 1: 理解上下文

在审查之前，理解代码背景：

- 确认编程语言和框架
- 理解代码的业务目的
- 识别受影响的模块
- 了解相关约束条件

### Step 2: 逐项审查

按照审查维度系统化检查：

```
for each 审查维度:
    for each 代码块:
        检查是否符合标准
        记录问题（如有）
        评估严重程度
```

### Step 3: 分类问题

将发现的问题分类：

| 严重程度 | 说明 | 处理建议 |
|---------|------|---------|
| 🔴 严重 | 必须修复，阻塞合并 | 立即修复 |
| 🟡 警告 | 应该修复，但不阻塞 | 近期修复 |
| 🔵 建议 | 可以改进，优化空间 | 有时间再改 |
| ⚪ 信息 | 仅供参考，无问题 | 知悉即可 |

### Step 4: 生成报告

输出结构化审查报告：

```
代码审查报告

📋 概览
- 文件数：N
- 代码行数：M
- 审查时间：...

🔍 审查结果
- 严重问题：X 个
- 警告：Y 个
- 建议：Z 个

📌 详细问题
[按严重程度列出]

✅ 优点
[列出做得好的地方]

📝 总体评价
[总结性意见]
```

## 注意事项

- 审查要客观，基于标准而非个人偏好
- 先理解代码意图，再评价实现方式
- 严肃问题要明确指出，改进建议要温和
- 平衡指出问题和认可优点
- 提供具体可操作的建议

## 🔴 严重问题（必须修复）

### SQL 注入漏洞
- **位置**：第 2 行
- **问题**：使用字符串拼接构造 SQL，存在 SQL 注入风险
- **风险**：攻击者可通过 user_id 执行任意 SQL，窃取/篡改/删除数据
- **建议**：使用参数化查询

```python
# 修复后
def get_user(user_id):
    sql = "SELECT * FROM users WHERE id = %s"
    result = db.execute(sql, (user_id,))
    return result[0]
```

### 数组越界风险
- **位置**：第 4 行
- **问题**：直接访问 result[0]，未检查结果是否为空
- **风险**：当 user_id 不存在时抛出 IndexError
- **建议**：添加空值检查

```python
# 修复后
def get_user(user_id):
    sql = "SELECT * FROM users WHERE id = %s"
    result = db.execute(sql, (user_id,))
    if not result:
        return None
    return result[0]
```

## 🟡 警告（应该修复）

### 返回类型不一致
- **位置**：第 4 行
- **问题**：找不到用户时抛异常，找到时返回数据
- **影响**：调用方难以统一处理
- **建议**：统一返回 None 或空对象，或抛出自定义异常

## 🔵 建议（可选改进）

### 函数命名可更具体
- **位置**：第 1 行
- **现状**：get_user 名称通用
- **建议**：get_user_by_id 更明确
- **收益**：提高代码可读性

## ✅ 优点

- 函数职责单一，逻辑清晰
- 代码简洁易读

## 常见问题速查

### SQL 注入
```python
# ❌ 错误
sql = "SELECT * FROM users WHERE id = " + user_id

# ✅ 正确
sql = "SELECT * FROM users WHERE id = %s"
cursor.execute(sql, (user_id,))
```

### XSS 漏洞
```javascript
// ❌ 错误
div.innerHTML = userInput

// ✅ 正确
div.textContent = userInput
// 或使用 DOMPurify.sanitize(userInput)
```

### 密码硬编码
```python
# ❌ 错误
DB_PASSWORD = "password123"

# ✅ 正确
DB_PASSWORD = os.getenv("DB_PASSWORD")
```

### 资源未释放
```python
# ❌ 错误
f = open("file.txt")
content = f.read()

# ✅ 正确
with open("file.txt") as f:
    content = f.read()
```
