# OpenSquilla 0.5.0 Preview 4

## Overview

OpenSquilla 0.5.0 Preview 4 is the supported opt-in macOS and Linux TUI release
candidate. The release installer pairs the platform-neutral Python package with
a self-contained OpenTUI host matching macOS or Linux and the machine's arm64
or x86_64 architecture. Use `opensquilla chat --ui tui` to try the full-screen
client; bare `opensquilla chat` remains on the plain rescue renderer during
this RC. This preview also adds safe Desktop profile recovery and explicit
whole-profile imports from another CLI/Desktop profile or historical Windows
Portable data.

The TUI and Web UI project the same Gateway-owned sessions, turns, history,
queues, usage, and approval decisions. Web UI remains the control plane for
configuration and multi-session supervision, while the TUI is the focused
client for the current terminal task.

## ✨ What's New

### Supported opt-in macOS TUI and Linux TUI

- The release includes `opensquilla-tui-host` companion wheels for macOS and
  Linux on arm64 and x86_64. The host is self-contained and normal installs do
  not require Bun, Node, a source checkout, or a first-run binary download.
- `opensquilla chat --ui tui` requires the matching host and fails clearly on a
  missing, incompatible, or wrong-architecture companion. `--ui plain` remains
  the minimal rescue surface, and explicit `--ui auto` can exercise the final
  startup policy.
- Full-screen shutdown restores the terminal for normal exit, interrupt, host
  failure, and Gateway disconnect, then prints one resumable session receipt.
- On both supported platforms the installed TUI can resume hydrated attachment
  history, project Web-originated turns, converge shared approvals, cancel
  queued work, and return a usable shell after leaving full-screen mode.

### Shared Gateway sessions

- Resuming a session hydrates canonical history rather than starting a separate
  terminal transcript. Session replacement operations update the visible
  transcript atomically.
- Turns, streams, tool activity, usage, queues, completion, and approval state
  can be projected across Web UI and TUI. Approval resolution is identified by
  the shared request id, so the first valid resolution closes the other client.
- The composer supports workspace references, local file uploads, image input,
  Gateway-visible paths, and attachment-only messages with explicit staging
  status before send.

### Safe profile recovery and imports

- Before starting a writable runtime, Desktop checks whether the active profile
  is safe. An uncertain or interrupted profile is held closed until the user
  chooses a trusted repair, restores a recorded backup, or completes recovery.
- A user may explicitly select another CLI/Desktop profile or historical
  Windows Portable data as the source for a complete profile copy. OpenSquilla
  never auto-selects the source; the source remains unchanged and the two
  profiles do not synchronize after import.
- An existing target is not silently merged file-by-file or chat-by-chat. The
  user keeps it as-is or makes a complete backup before replacing it.
- Interrupted profile imports and settings transactions can safely resume,
  roll back, or complete. The copied profile is verified before restart, and
  imported scheduler jobs remain paused.

### Packaging and recovery

- macOS and Linux install, upgrade, reinstall, and rollback commands keep the
  core and companion on the same `0.5.0rc4` release. Product, host, and protocol
  version mismatches are rejected before full-screen startup. Rolling back to
  an older core-only release removes the newer companion rather than mixing
  versions.
- Host startup uses an authenticated loopback connection and a versioned ready
  handshake. A runtime crash exits after terminal recovery rather than changing
  renderers in the middle of a turn.
- Native Windows TUI remains a separate follow-up release for its ConPTY host;
  it reuses the same session and protocol contracts. WSL follows the Linux
  support path.

- Linux x86_64 TUI host:
  `opensquilla_tui_host-0.5.0rc4-py3-none-manylinux_2_28_x86_64.whl`

  `opensquilla_tui_host-0.5.0rc4-py3-none-manylinux_2_28_aarch64.whl`

The packaged Linux TUI host requires glibc 2.28 or newer. Native musl/Alpine is
not included in this RC; WSL follows the Linux package path.

No Windows portable assets are published for 0.5.0 preview releases. In
particular, there is no 0.5.0rc4 portable zip or portable latest alias. The
macOS `.zip` listed above is the Electron updater artifact, not a portable
distribution.

The macOS app and TUI host release artifacts are signed and notarized. The
Windows installer remains unsigned; review the
[code signing policy](https://github.com/opensquilla/opensquilla/blob/v0.5.0rc4/docs/code-signing-policy.md)
before downloading it in a managed environment.

## Upgrading from Preview 3

Preview 4 is a new release at `v0.5.0rc4`; it does not reuse or move the
`v0.5.0rc3` tag.

Windows Desktop users must install Preview 4 directly over the existing
installation. **Do not uninstall Preview 3 first**: its uninstaller may remove
`%APPDATA%\OpenSquilla`. Back up that directory before upgrading.

macOS and Linux users can install the matching paired wheels automatically
with:

```sh
curl -LsSf https://opensquilla.ai/install.sh | bash
```

The equivalent fully pinned Apple Silicon macOS command is:

```sh
uv tool install --python 3.12 \
  --with "opensquilla-tui-host @ https://github.com/opensquilla/opensquilla/releases/download/v0.5.0rc4/opensquilla_tui_host-0.5.0rc4-py3-none-macosx_11_0_arm64.whl" \
  "opensquilla[recommended] @ https://github.com/opensquilla/opensquilla/releases/download/v0.5.0rc4/opensquilla-0.5.0rc4-py3-none-any.whl"
```

Intel macOS uses the sibling x86_64 companion wheel. Linux uses the
`manylinux_2_28_x86_64` or `manylinux_2_28_aarch64` companion matching its
architecture. Keep both wheel URLs on the same tag when upgrading or rolling
back. Configuration formats from every released OpenSquilla version remain
supported; back up the active home before an upgrade if you may need to return
to Preview 3.

## Container deployment

The multi-architecture gateway image is published as:

```sh
docker pull ghcr.io/opensquilla/opensquilla:v0.5.0rc4
```

Docker `latest` follows the most recently pushed release tag. Pin
`v0.5.0rc4` when reproducibility matters.

## Privacy and network controls

OpenSquilla does not add anonymous TUI runtime telemetry. To disable automatic
install telemetry, passive release checks, and desktop startup update checks,
set:

```sh
OPENSQUILLA_PRIVACY_DISABLE_NETWORK_OBSERVABILITY=true
```

or configure:

```toml
[privacy]
disable_network_observability = true
```

The compatibility variables `OPENSQUILLA_TELEMETRY_DISABLED=true` and
`OPENSQUILLA_UPDATE_CHECK_DISABLED=true` remain honored. Manual
user-initiated release and download actions may still contact GitHub. See the
tag-pinned [`PRIVACY.md`](https://github.com/opensquilla/opensquilla/blob/v0.5.0rc4/PRIVACY.md)
and [`THIRD_PARTY_NOTICES.md`](https://github.com/opensquilla/opensquilla/blob/v0.5.0rc4/THIRD_PARTY_NOTICES.md).
